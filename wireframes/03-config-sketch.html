03 — CONFIG ARCHITECTURE

Config Architecture

How the per-status config drives card + detail without bespoke conditionals

The problem it solves

Currently, status-specific display logic is scattered: isIntakeRejected(), needsRework(), isSidebranch() computed signals each add conditional branches. Every new column-specific behaviour adds another condition.

A single config object per grooming status means: one place to define what a status looks like. Card, detail, and action bar all read from it. Adding a new column = adding one config entry.

Config interface (TypeScript sketch)

interface ColumnConfig {
  // Card
  primaryContent:
    | 'rejection_callout'   // intake_rejected
    | 'rework_reason'       // needs_rework
    | 'source_attribution'  // ungroomed
    | 'body_excerpt'        // default
  showTags: boolean
  showOwnerChip: boolean

  // Card actions (in order)
  cardActions: CardAction[]

  // Detail
  defaultTab:
    | 'overview'
    | 'body'
    | 'activity'
    | 'thread'

  // Detail sticky bar
  primaryAction: ActionDef
  secondaryAction: ActionDef
  // overflow: always shows full set
}

Config entries per status

Status Primary content Card actions Default tab Sticky bar
intake_rejected Boris's question delete / edit / archive thread delete / edit
needs_rework Rework reason + target fix / reassign / delete overview edit / reassign
ungroomed Source attribution → note / delete overview triage / delete
classified body_excerpt decompose / edit body decompose / edit
decomposed body_excerpt mark ready / edit overview mark ready / edit
ready body_excerpt edit / archive overview send to exec / edit

intake_rejected is the first fully-designed row. Others are placeholder logic — correct the table as each column gets attention.

Touch target requirements

Card action buttons: min 44px height
Current card__action-btn: no explicit height set — likely ~32px.
All three buttons share equal width (flex: 1) within card__actions row.
Padding: 10px top/bottom in wireframe → ~44px with 12px font line-height.

Sticky bar buttons: min 48px height
Bigger surface = less precision needed when scrolling stops.
Primary action: flex: 2 (wider, harder to miss).

Tab strip: min 44px height
Currently ~36px. Worth increasing on mobile breakpoint.

Where the config lives

// src/app/features/grooming/
//   grooming-column.config.ts

export const COLUMN_CONFIG:
  Record<GroomingStatus, ColumnConfig> = {
  intake_rejected: { ... },
  needs_rework: { ... },
  ungroomed: { ... },
  ...
}

Consumed by:

// grooming-card.ts
readonly config = computed(() =>
  COLUMN_CONFIG[this.item().grooming_status]
)

// vault-item-detail-body.ts
readonly config = computed(() =>
  COLUMN_CONFIG[this.item()?.grooming_status]
)
readonly activeTab = signal(
  this.config()?.defaultTab ?? 'overview'
)

Implementation sequence

Step 1 — Create grooming-column.config.ts with the interface + intake_rejected entry.

Step 2 — Grooming card reads config: swap action row content, show Boris's question as primary content for intake_rejected.

Step 3 — Detail body reads config: initialise activeTab from config.defaultTab.

Step 4 — Sticky action bar in detail: new component DetailActionBar that receives the config's action definitions.

Step 5 — Fill in config entries for remaining statuses as needed (not all at once).
