Still aspirational: rationale alongside AI scores. When priority_scored activity events carry "P2 because deadline +3d and dependency X", this chip gets a hover with the reasoning. Whiteboard row 42.
5 Activity log — structural facts only decision
Four event types: created, assigned, status_changed, thread_message_posted. The thread_message events are pointers — body lives in the Thread panel beside, not here.

Why both streams? Different queries. Activity answers "when did this move?"; Thread answers "what's being discussed?". Collapsing them loses the distinction (P15 · K14).
6 Thread panel — composed via <app-thread-view> insight
The VaultItem feature (future) will embed the Thread feature's top-level component. This prototype shows what the embedded result looks like: messages rendered with kind-specific styling, open questions pilled, composer at the bottom.

Implementation: features/thread/components/thread-view — built in parallel, used here. No duplication.
7 due_at / completed_at — real-world dates decision
Two new nullable timestamps on VaultItem. due_at captures real-world deadlines (not invented from priority). completed_at mirrors status='done' for cheap reads ("what did we finish this week?") — kept in sync with the paired status_changed → done activity event by convention.

Urgency display ("in 3 days", or red when overdue) is a projection, not stored. Styling a due date as red because it's past is a UI rule; the data is just an ISO string.
8 Blocking dependencies — junction + conditional readiness check decision
Junction: vault_item_dependencies(blocker_id, blocked_id). Semantics: "the blocker must be in state 'done' before the blocked item can move past readiness".

Readiness gains a new check (unresolved_blockers) that only surfaces when an item has blockers — same pattern as open_questions. Items without dependencies don't see the check.

MVP has only one dependency kind (blocks). Future kinds (relates_to, duplicate_of) are deferred — add when a real case demands them.
9 source — origin tracking decision
source: { kind, ref, url? } | null on VaultItem. Kinds: manual | email | telegram | agent | url | pr-comment. ref is an opaque id from the origin system (gmail message id, telegram id, PR number, actor slug). Optional clickable URL when meaningful.

Designing this in now (while types are cheap) so the email intake layer (whiteboard row 21) can populate it without a shape change later. UI shows a small badge near the top; rarely the focus, but always available.
10 Actionability — the intake-quality verdict decision
actionability: 'clear' | 'needs-breakdown' | 'vague' | null. Set by the hermes/intake-quality skill. 'vague' means the body is too sparse for agent work; intake posts clarifying questions and moves the note to intake_rejected.

Grooming vocabulary change. Old statuses (analysing / questions / proposed) were present-progressive — implying in-flight. But in-flight is tracked on dispatch_queue, not the note. New vocabulary names completion states: ungroomed → intake_rejected / intake_complete → classified → decomposed → ready.

questions_pending is gone as a status — open questions are an orthogonal blocker via ThreadMessage, surfacing independently in readiness regardless of pipeline position.
11 Classification — rationale + confidence decision
Three new fields written by hermes/vault-classify (the split-out skill that runs after intake passes):
• ai_rationale: string | null — one-sentence why
• priority_confidence: number | null — 0.0–1.0 explicit uncertainty
• actionability: Actionability | null — set by intake, read by everyone

In this fixture, vault-classify hasn't run yet because intake rejected the body. So rationale and confidence are null — the UI shows the dashes honestly.

Skill split rationale (P1 — micro-skill decomposition): `vault-analyse` was doing 5 jobs (priority, confidence, actionability, tags, rationale + optional questions). Split into intake-quality (budget tier, actionability + questions only) and vault-classify (standard tier, priority + confidence + rationale + tags). Cheaper aggregate — bad intake bounces in Haiku-land before paying for classification.

Full hermes-side design: docs/architecture/hermes-skill-alignment.md.
