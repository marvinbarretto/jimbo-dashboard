## What qualifies as a LocalShout event

- Venue must be in Hertfordshire (any district)
- Public event — ticketed or free, but not private bookings or invite-only
- Has a date/time in the next 6 months
- Has a discoverable web page or booking URL
- NOT: gym classes, religious services, business networking

## Target audience

Locals aged 25-55 looking for weekend/evening plans.

## Coverage priorities

1. Hertford, St Albans, Watford (tier 1)
2. Hitchin, Welwyn, Stevenage (tier 2)
3. Everything else (tier 3)

## Known source types

- Council listings pages
- Venue websites (pubs, community halls, theatres)
- Facebook events (if public)
- Local press (Herts Advertiser etc.)

1 Criteria as freeform markdown — enough? question
Currently a single markdown string. Agents read it whole and reason over it. Simple, flexible, easy to edit.

Alternative: structured breakdown (goals, rules, constraints, exclusions, source list) as sub-entities. Would make projections easier ("show me all exclusion rules across projects") but rigid.

My lean: markdown for MVP. Extract structure only when a concrete need forces it (e.g. a relevance-classifier needs to consume rules programmatically, not via prompt-stuffing).
2 repo_url — who uses it, how? decision
Per principle P9, project-scoped skills live in the project's repo. The dashboard's skill catalogue is an *index*, not the source of truth.

How discovery works: jimbo (or a scheduled job) clones/pulls each active project's repo, scans for skills/, and indexes what it finds. Dashboard displays the union of global + discovered skills.

Open: what does jimbo do if a project's repo is unreachable? Show stale-indexed skills with a "last synced 3d ago" warning? Or hide them until re-synced? Leaning toward *show with warning* — operator needs to know what's there.
3 Skill namespacing via slug prefix insight
No new column needed on the skills table. The slug itself says where it belongs:

• intake-quality → global (no prefix)
• localshout.event-qualifier → project-scoped (prefix matches project id)

UI groups by parsing the prefix. Filesystem mirrors: hermes/skills/intake-quality/ vs localshout/skills/event-qualifier/.

Collision rule: a project cannot shadow a global skill name. localshout.intake-quality would be valid; intake-quality inside localshout/skills/ wouldn't be.
4 Stats and item list — projection or stored? decision
Stats (12 active, 3 ungroomed, etc.) are obviously derived: COUNT(vault_items WHERE project_id = 'localshout' AND ...). Don't store.

The item list here is a paged view over vault_items filtered by project_id. Same data as the main vault screen, different filter.
5 Owner as ActorId — supports jimbo-spawned projects decision
owner_actor_id: ActorId. Most projects today = @marvin. Future: @jimbo can own weekend prototypes it initiates ("why don't you prototype something?").

Prototypes are just lightweight projects — same shape, most fields null (no repo_url, brief criteria, maybe one or two vault items before archive). The Project entity scales from "full SaaS product" down to "jimbo's weekend experiment" without needing a separate prototype type.
