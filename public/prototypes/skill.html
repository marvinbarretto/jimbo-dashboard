1 Slug-prefix namespacing · slash separator decision
Format: {project-id}/{skill-name}. Always prefixed — no "bare = global" shortcut. Every skill belongs to exactly one project; the core project is named hermes.

Slash (not dot) because: (a) OpenRouter model IDs already use slash — consistent, (b) maps directly to filesystem at hermes/skills/intake-quality/, (c) standard for namespaced resources (Docker tags, OCI, URL paths).

One level only. skillNamespace() splits on first slash. Sub-grouping (localshout/scraping/event-qualifier) isn't supported. If a project grows that complex, spin a sub-project.
2 Skill version vs prompt version — deferred with tradeoffs clarified question
Prompts are versioned (prompt_versions). Skills reference the current prompt version by id. If the prompt's wording evolves (v2 → v3), the skill's behaviour changes but the contract (schemas) doesn't.

Open question only fires when the contract itself changes — e.g. input_schema gains a required field. Two paths:

(a) New slug on breaking change — hermes/intake-quality-v2 is a separate Skill row.
✓ Simpler structure — no extra tables.
✓ Old callers keep working on the old slug until migrated.
✗ Catalogue bloats: after 5 evolutions you have 5 slugs for "the same thing".
✗ Benchmark history fragmented — each variant has its own history.
✗ actor_skills must be re-granted for every new variant.

(b) Version the skill itself — add skill_versions table. Skill row points at current_version_id; schemas and prompt_id move onto the version row.
✓ Catalogue stays clean — one row per skill concept.
✓ Actor grants + benchmark history attach to the skill, not the variant.
✓ Callers can pin to a specific version.
✗ Extra table, extra complexity.
✗ Mirrors prompt versioning, but skills version less often — the overhead/benefit ratio is worse.

We don't have to decide now. No schema has actually broken yet. Deferred to whiteboard with both options documented. Revisit when the first skill's schema actually changes breaking — likely months away.
3 Model tier — reusing existing enum · OpenRouter-inspired decision
Dropped -class suffix. model_hint now uses the existing ModelTier from features/models/utils/model.types.ts: free | fast | balanced | powerful. Mapping against current actors:
• free → Ralph (local qwen, no token cost)
• fast → Boris on Haiku (cheap hosted, quick)
• balanced → Boris on Sonnet (default hosted)
• powerful → Boris on Opus (heavy reasoning)

OpenRouter inspiration. OpenRouter doesn't have tiers — they expose raw specs (params, price, context, modality) and let filters pick. We keep coarse 4-tier for now because operator-facing routing decisions don't benefit from finer granularity. When we need it, we can add richer fields (min_params, max_cost_per_run, etc.) — no migration needed, just additive columns.

Capabilities (vision, long-context, tool-calling) remain separate from tier — deferred to capabilities: string[] when first needed (whiteboard row 47).
4 Repo is source of truth · cache displayed clearly decision
Source of truth lives in the repo at {project.repo_url}/skills/{name}/*. The dashboard stores a cache for fast reads. UI exposes the cache nature explicitly:

• last_indexed_at field on every skill row
• "synced 19h ago" badge in the header and next to schema blocks
• Unreachable repos surface as "last successful sync 3d ago — check repo_url"
• Stale rows (say, >24h) get a soft yellow flag

Editing model: operators don't edit skills in the dashboard. To change a skill, edit the files in the repo and re-index. Dashboard's role is read/observe, not write.

The form fields visible in this prototype are therefore slightly misleading — they'd be read-only in production. Flagging now so we get the UI right when implementing.
5 source_repo — FK to Project, not free-text decision
Changed from string to ProjectId. Joins come for free. Slug prefix + FK must match — enforced at write time. Two representations of the same fact, each serving a purpose: prefix for uniqueness + readability, FK for joins.

Invariant: skillNamespace(id) === source_repo. Anything else is a data-integrity bug.
6 Pure-prompt vs tool-using skills insight
intake-quality uses no tools — pure prompt → verdict.

Compare to extract-events-from-html (shown in related): prompt_id = null, no LLM, deterministic parser. Or fetch-url: prompt_id = null but uses a web-fetch tool.

Three shapes of skill:
• Pure prompt: prompt_id set, no tools, LLM does the work
• Tool-augmented prompt: prompt_id set + tools declared, LLM uses tools mid-reasoning
• Deterministic code: prompt_id = null, tools optional, no LLM invocation — just runs

Current shape supports all three. Good.
7 Deactivation cascade behaviour question
If is_active = false on a skill, what happens to the actor_skills rows referencing it?

Options:
• Preserve: junction rows stay; skill just drops out of routing. Reactivation is one field flip.
• Cascade delete: junction rows disappear; reactivating a skill requires re-granting to actors.

Lean: preserve. Deactivating a skill should be reversible without losing operator intent about which actors were trusted. The dispatcher already filters by skill.is_active when picking; junction presence is harmless.
