Discussion · Questions · Issues

1 Three kinds: comment / question / answer decision
comment for unstructured back-and-forth, question for things that expect an answer and block until they get one, answer for resolutions that point at a specific question via in_reply_to.

Three is deliberately the ceiling. Adding result for agent skill-output was considered and rejected — agent results are structured enough to deserve their own activity event type (skill_invoked, future), not a conversational message kind.
2 Open questions block dispatch insight
The two open-pilled questions each add a failing readiness check (open_questions) — until resolved, the item can't move to ready. Readiness only surfaces this check when questions exist; items that never went through intake-quality don't see it.

This is what gives the gatekeeper teeth. The skill produces questions; the questions become dispatch-blockers via the existing readiness machinery. No new state machine needed.
3 Answer ↔ Question via in_reply_to + answered_by decision
Both pointers exist because they serve different reads:
• answer.in_reply_to = question_id — "this answer addresses that question"
• question.answered_by = answer_id — "this question is resolved by that answer"

Redundant? A little. But querying "is this question resolved?" via a single field on the question row is cheap. Querying "what did this answer answer?" via a single field on the answer row is cheap. Without both, one direction requires a JOIN.

Mutation: when an answer is posted with in_reply_to = question_id, the system sets question.answered_by = answer_id in the same transaction. The two stay consistent.
4 Agents posting — free-form content, structured behaviour insight
Agents (jimbo, boris, ralph) post messages the same way humans do — kind + body. The distinction is in author, not message shape. Boris's "standing by" comment is structurally identical to a marvin comment.

What this enables: a "skill run completed" agent comment can be written by the agent after finishing work, and it reads naturally in the conversation. No separate "agent output" entity needed for MVP.

If agent output ever gets structured enough to deserve its own shape (e.g. tool-calling traces), that becomes a dedicated activity event (skill_invoked), not a new message kind.
5 Composer — two or three writes per post decision
One operator action ("post message") writes:
1. thread_messages — the content
2. activity_events — the audit fact (thread_message_posted with message_id pointer)
3. attachments × N — one row per attached image/file (if any)

All in the same transaction. Keeps P6 intact (every mutation produces an event). Activity timeline shows "who posted what kind"; Thread view shows the full body + any attachments.

Denormalising message_kind onto the event row lets "activity timeline" views render without a JOIN to thread_messages — useful for the Stream surface.
6 Attachments — attached to messages, not items decision
Attachment is a separate entity with an FK to ThreadMessage. Single home avoids polymorphic columns (attach-to-item vs attach-to-message). VaultItem-level "attachments" are thread messages posted at creation — intake comment with a screenshot.

Two workflows this enables:
• Operator drags an image into thread — browser uploads to jimbo-api, gets URL, POSTs an Attachment record linked to the composing message.
• Agent takes a screenshot via Playwright — agent runs the browser, captures, uploads to jimbo-api, posts a thread message with the Attachment linked.

External refs stay inline. A GitHub PR link is a URL in the message body, not an attachment. Attachments are for bytes we host; pointers to external artefacts are prose.
